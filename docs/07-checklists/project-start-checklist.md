# Project Start Checklist

Use this checklist when a team is about to introduce structure into a new feature or module.

- [ ] Can we describe the real problem without naming a pattern first?
- [ ] Which project layer should own this responsibility?
- [ ] Is the variation already real, or only expected later?
- [ ] Would a smaller refactor solve the problem first?
- [ ] Do we know what `When Not to Use` looks like for the chosen approach?
- [ ] Are external APIs or vendor SDKs isolated from business language?
- [ ] Can the design be tested without real infrastructure?
- [ ] Are we avoiding empty scaffolding and speculative files?
