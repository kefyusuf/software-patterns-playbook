# Pattern Review Checklist

Use this checklist in code review, architecture review, or documentation review before accepting a pattern recommendation.

- [ ] Does the document start from the problem instead of a textbook definition?
- [ ] Is the recurring pressure visible and concrete?
- [ ] Does the recommendation include trade-offs?
- [ ] Does it include `When Not to Use` guidance?
- [ ] Does it mention a simpler alternative?
- [ ] Is the pattern placed in the right project layer?
- [ ] Are related patterns and likely confusions explained?
- [ ] Would another developer understand the decision without reading internal implementation details?
